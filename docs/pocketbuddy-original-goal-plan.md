# PocketBuddy 原始目标差距与推进计划

本文不再沿用阶段清单，而是直接按 PocketBuddy 的原始目标来判断：

> 把网页、论文、文章和脑中的想法，转成笔记、记忆、图片、图谱和可执行的小产品方案。

## 一句话结论

当前工程已经不是“壳子”，核心插件链路是通的，能完成基本的投喂、发明、归档、观察和设置。

但如果按原始目标的完整形态看，它还没有完全达标。现在更像一个“本地优先、有人在环的成长型 Agent”，还不是一个真正具备持续记忆、主动学习、强语义召回和稳定多步执行能力的完整 PocketBuddy。

## 补充后的最终形态

这张示意图把 PocketBuddy 的最终目标说得更清楚了：它不是普通的问答型 AI 插件，而是一个**内容加工 Agent**。

它的核心不是“回答问题”，而是把输入材料变成可持续复用的知识资产：

- 输入：想法、网页、论文、文章、划词片段
- 处理链路：`想法 -> 规划 -> 调研 -> 反思 -> 信息编排 -> 生成`
- 中间产物：笔记、记忆候选、Archive Note、知识图草案、产品灵感、MVP 草图
- 最终产物：知识密集型图片、脉络图谱、知识卡、可回看的个人资产

这意味着 PocketBuddy 的目标要再补三层：

1. **加工链路要显式化**
   - 后台不是“拿到输入就直接出图”，而是先规划、再调研、再反思、再编排，最后才生成。
   - 每一阶段都要有可观测状态，前台要能看到正在发生什么。

2. **产物要资产化**
   - 结果不能只是一次性回答，而要变成可保存、可回看、可再编辑、可继续生长的知识资产。
   - 图片、图谱、笔记、记忆候选之间要互相可追溯。

3. **界面要仪表盘化**
   - 少文字说明，多状态、多图表、多过程动画。
   - 前台主要负责展示“输入 -> 加工 -> 输出”的过程感，而不是堆叠解释文本。

补到这里以后，PocketBuddy 的定位就更完整了：

- 它是一个会读的 Agent
- 也是一个会记的 Agent
- 更是一个会把信息加工成高密度知识图像的 Agent

## 目前已经完成的核心能力

### 1. 前端与交互面

- Side Panel 已经包含 `创意 / 喂养 / 归档 / 观察 / 设置` 五个区域。
- 用户可以从界面直接发起想法生成、页面读取、分析、归档、图片生成、图谱生成、快照回滚、画像编辑。
- 交互状态、错误提示、历史列表、版本对比都已经有明确承载位置。

### 2. 消息与调度链路

- 前端通过强类型消息发给 background。
- background 统一路由到对应处理函数。
- content script 负责页面/选区提取。
- 这条链路已经闭合，并且能在真实浏览器环境里跑通。

### 3. 页面理解与喂养

- 已经能读取当前网页内容、做结构化分析、产出 note card 和 memory candidates。
- 已经支持选区捕获。
- 已经支持把网页内容保存为页面上下文和归档笔记。

### 4. 创意生成

- 可以把一句话想法收束为产品概念。
- 可以生成图片 Prompt。
- 可以给出 MVP 计划和下一步任务。
- 反馈按钮可以影响后续画像和补丁记录。

### 5. 记忆与观察

- 本地记忆、画像、历史、候选记忆、长期记忆、产物、图片、图谱、快照都已入库。
- 观察页已经能展示时间线、版本对比、快照差异、补丁历史。

### 6. 模型与适配层

- LLM 已有 mock / real 的分层。
- 图片生成也有 mock / proxy 的分层。
- 图谱生成、归档笔记、页面分析都已经形成清晰的服务层。

## 还没有真正达标的地方

### 1. 还不是“强语义记忆型 Agent”

当前召回主要是启发式排序，不是真正的语义检索。

现状特征：
- 通过关键词、近期性、偏好标签做召回。
- 能“看起来相关”，但还不足以稳定做到“记得准、找得准、串得准”。

这意味着：
- 它能帮你回忆一些线索。
- 但还不能像真正的长期记忆系统那样，把知识、上下文、偏好和产物之间的关系稳定地编织起来。

### 2. 还不是“会主动工作的 Agent”

当前大多数行为仍然是用户点击驱动。

现状特征：
- 用户点“读取”“分析”“生成”“保存”“回滚”。
- Agent 在收到指令后执行，但没有明显的自主任务规划、主动提醒、持续跟踪或自发再行动作。

这意味着：
- 它是一个好用的助手。
- 但还不是一个会自己推进任务的持续型 Agent。

### 3. 道具体系还不是真正的“可编排工具系统”

现在的 `gadget` 更像一组领域 helper。

现状特征：
- `IdeaLens`、`ProductCamera`、`ShrinkLight`、`MemoryBread`、`AnywhereDoor`、`TimeNotebook` 等命名和职责清晰。
- 但它们还没有形成真正的动态工具市场，也没有工具注册、工具权限、工具版本、工具自举、工具试运行这样的完整体系。

这意味着：
- “道具概念”已经成立。
- “道具系统”还没有完全成立。

### 4. 输出适配还不够丰富

目前主要输出是：
- 文本概念
- 图片请求
- 图谱记录
- 归档笔记

但原始目标里“可执行的小产品方案”还可以进一步扩展为：
- markdown / 文件导出
- 外部笔记系统导出
- 更标准化的项目草案
- 更完整的图片/图谱/笔记互导

### 5. “能观察自己变化”还不够强

观察页已经有历史和对比，但“因果链”还不够深。

现状特征：
- 能看到喂养、画像变化、产物、反馈。
- 但还不能非常可靠地回答：是哪一段输入，具体如何改变了哪一个偏好、哪一种输出风格、哪一个工具选择。

### 6. 真实模型验证还不够

从当前仓库里能看到：
- 代码支持真实 LLM / 图片代理。
- 默认配置是 mock。
- 已有的 e2e 脚本验证的是本地链路和注入链路，不是 live model 的真实回归。

也就是说：
- 我没有看到足够证据说明真实模型已经被稳定测试过。
- 这部分需要单独补一轮“真人模型接入验证”。

## 核心组件完成度判断

| 组件 | 完成度 | 判断 |
|---|---:|---|
| Frontend / Side Panel | 高 | UI 已经成型，功能入口明确 |
| Messaging Layer | 高 | 强类型消息链路清晰 |
| Background Service Worker | 高 | 路由、调度、存储、注入都已接上 |
| Content Script | 中高 | 能用，但仍受页面类型和权限边界影响 |
| Agent Core | 中高 | 已能编排，但自治度不足 |
| Gadget System | 中 | 名字齐、职责齐，系统化程度不足 |
| Page Intelligence | 高 | 已能读取、分类、分析、提炼 |
| Memory Layer | 高 | 数据层完整，但检索能力不够强 |
| Media / Output Adapters | 中 | 图片、图谱有了，导出和外接还不够 |

## 目前最关键的缺口

如果只抓最影响“原始目标达成”的缺口，优先级大概是：

1. 语义级记忆与召回
2. 主动式任务编排
3. 真模型验证与回归
4. 输出适配扩展
5. 更深的观察与因果追踪
6. gadget 的系统化与可扩展化

## 接下来怎么推进

下面按“先把底座做准，再把成长闭环做深”的顺序推进。

### 阶段 1：先确认真实模型链路真的可用

目标：
- 验证 live LLM 能返回稳定结果。
- 验证 live image proxy 能正常工作。
- 验证失败时能优雅降级到 mock 或可解释错误。

要做的事：
- 准备一份最小真实配置。
- 用最小想法、最小页面、最小图片请求做 smoke test。
- 记录成功和失败样本。

验收标准：
- `llmProvider !== 'mock'` 时，真实请求能稳定返回。
- `imageMode === 'proxy'` 时，真实图片代理能稳定返回。
- 失败时不会把 side panel 卡死。

### 阶段 2：把记忆从“列表”升级成“关系网”

目标：
- 让 recall 更像“找关系”，而不是“拼关键词”。

要做的事：
- 给记忆增加更强的结构化标签。
- 建立主题、上下文、产物、反馈之间的关联边。
- 引入更强的检索策略，至少做 hybrid recall。

验收标准：
- 同一个用户主题在多次输入后，召回结果更稳定。
- 能解释“为什么这条记忆被召回”。

### 阶段 3：把 Agent 从“响应式”推进到“半主动式”

目标：
- 让 Agent 能基于历史和状态给出下一步建议，而不是只等点击。

要做的事：
- 增加任务状态机。
- 增加下一步建议和提醒机制。
- 让它在关键状态变化后自动提出建议动作。

验收标准：
- 用户完成一次阅读或发明后，Agent 能主动给出下一步建议。
- 建议和当前记忆、画像、产物能对应起来。

### 阶段 4：把输出适配做完整

目标：
- 让结果不止停留在插件内。

要做的事：
- 增加 markdown / 文本 / 文件导出。
- 增加更标准的笔记导出。
- 梳理图片、图谱、笔记、概念之间的互导路径。

验收标准：
- 生成物可以稳定导出。
- 导出内容结构清晰，可复用。

### 阶段 5：把观察页变成真正的“演化仪表盘”

目标：
- 让用户看懂 agent 是怎么被喂出来的。

要做的事：
- 增加更清晰的因果链。
- 记录每次喂养对画像和产物的影响。
- 给反馈、补丁、回滚和版本变化建立更强关联。

验收标准：
- 用户能从观察页看懂“输入 -> 变化 -> 输出”。
- 能定位某次改变是由哪次喂养或反馈触发的。

### 阶段 6：把 gadget 系统真正系统化

目标：
- 从“命名可爱的一组 helper”升级成“可扩展的道具体系”。

要做的事：
- 增加工具注册与描述能力。
- 增加工具能力边界和权限边界。
- 增加工具试运行和版本记录。

验收标准：
- 新道具可以按统一协议接入。
- 道具能被 agent core 识别、调用和观察。

## 建议的验证顺序

1. 先做真实模型 smoke test。
2. 再做 recall / memory 的准确性验证。
3. 再做一次完整的“网页输入 -> 产物输出 -> 记忆变化 -> 观察页回放”的端到端验证。
4. 最后补导出和 gadget 系统化。

## 当前判断

如果以原始目标衡量：

- 核心骨架已经完成。
- 关键闭环已经可用。
- 但“强记忆、强自治、强可观察、强可扩展”这四个方向还没有完全达标。

所以现在不是从 0 开始，而是进入“把完整性补齐”的阶段。
