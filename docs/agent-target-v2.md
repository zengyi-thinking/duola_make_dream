# PocketBuddy 工程目标 v2

## 一句话目标
把 PocketBuddy 从“会生成内容的浏览器插件”升级成“可投喂、可观察、可自我调整的成长型 Agent 插件”。

这个版本不是单纯加功能，而是把产品重心从“创意产出”调整为“养成闭环”:
- 用户喂给它数据
- Agent 产出结果
- 用户观察它如何变化
- 用户再反向调教它的模型、人格、工具和风格

## 当前代码的真实状态

现在工程里已经有一批可复用的基础，不是从零开始：

- 前端已经有 `creative / reading / archive / observation / settings` 五个 tab。
- `background` 已经是统一消息路由中心，消息链路是通的。
- LLM、图片、页面提取、记忆、本地存储、反馈、harness patch 都已经有雏形。
- 头像组件和头像资产已经存在。
- 图片生成和 LLM 都已经支持 mock / real 的分层。

但当前实现还更像“功能集合”，还没有完全变成这张图想要的“成长系统”。

## 新需求解读

这张需求图实际上提出了四个核心页面，并且每一页的职责很明确。

### 1. 发明页面
目标是让用户把创意记录下来，并把它变成更清晰的图片和更完整的产品雏形。

这不是单纯的“输入想法生成结果”，而是：
- 记录灵感
- 形成产品概念
- 生成更清楚的图像表达
- 让创意有版本、有对比、有反馈

### 2. 喂养页面
目标是把网页里的信息当作“投喂资料”提供给 agent。

这意味着阅读页不应只是“读页面”，而是：
- 收集可用原材料
- 给 agent 输入知识、上下文和偏好
- 把网页变成可进化的记忆燃料

### 3. 观察页面
这是后续一定要补齐的状态面板，但在当前优先级里不应该抢占 P0。

它不只是“展示列表”，而是要能：
- 观察 agent 各组件状态
- 观察喂养数据如何影响人格画像
- 观察 harness / 工具 / 记忆如何演化
- 提供历史备份和回溯路径
- 在方向跑偏时，找到“最懂我们的那个版本”

### 4. 设置页
设置页不再只是配置项，而是 agent 的“出生地”和“身份实验室”。

它需要支持：
- LLM / 文生图模型配置
- 用户画像，也就是“大雄画像”
- agent 头像和风格配置
- agent 自己创作小工具并测试
- 人格、视觉和工具的初始设定

## 当前页面与新需求的对应关系

### 发明页
当前可复用：
- `CreativeTab`
- 想法输入
- 产品概念生成
- 图片 Prompt
- MVP 计划
- 用户反馈按钮

还缺的关键能力：
- 创意版本管理
- 图片清晰度/结果迭代
- 更强的“发明记录”沉淀

### 喂养页
当前可复用：
- `ReadingTab`
- 当前页读取
- 页面分析
- 选区捕获
- 记忆候选生成

还缺的关键能力：
- 明确的“投喂”入口和流程
- 资料来源分类
- 喂养后的知识沉淀与关联
- feed 结果和 agent 记忆的可追踪关系

### 观察页
当前可复用的基础：
- `ObservationTab` 里的状态卡片、时间线、备份和补丁历史
- `ArchiveTab` 里的笔记归档
- `MemorySummary`
- `HarnessPatch`

必须新增的能力：
- 组件状态面板
- 记忆演化时间线
- 画像变化历史
- harness patch 审查区
- 备份 / 回滚入口
- 观察“喂养 -> 变化 -> 产出”的因果链

### 设置页
当前可复用：
- LLM 供应商配置
- 图片供应商配置
- 清除数据
- agent 身份与头像配置
- 头像组件

必须新增的能力：
- 用户画像编辑
- agent 头像 / 风格选择
- 自定义工具创建与试运行
- 更明确的“身份实验室”定位

## 需要调整的工程目标

### 目标 1：从“会生成”升级为“会养成”
P0 先把三件事跑通：
- 设置页接通 LLM / 文生图模型
- 喂养页稳定采集网页内容并进入记忆
- 发明页稳定把想法转成可见产物

在此基础上，再逐步补齐“输入 -> 观察 -> 调整 -> 再输入”的成长闭环。

### 目标 2：从“结果页”升级为“状态系统”
后续要补的能力是：
- 观察 agent 如何被喂养
- 观察记忆如何变化
- 观察人格如何被塑造
- 观察工具如何被生成或修改

### 目标 3：从“配置项”升级为“身份控制台”
设置页应该承载：
- 模型
- 头像
- 风格
- 用户画像
- 工具实验

### 目标 4：从“静态记忆”升级为“可回溯演化”
现在 memory 更像本地数据库。
新目标要让它变成：
- 有历史
- 有版本
- 有恢复路径
- 有偏差可纠正

## 建议的优先级

### P0
- 设置页接通 LLM / 文生图模型
- 喂养页稳定采集网页内容并进入记忆
- 发明页稳定把想法转成可见产物

### P1
- 观察页展示基础状态、画像变化和关键历史
- 给设置页增加用户画像和头像风格配置

### P2
- 增加版本回滚
- 增加观察时间线和 harness patch 审查
- 增加创意和图片结果的版本对比
- 增加更强的知识召回和关联

## 需要保留的现有架构

这些部分不用推倒重来，应该继续保留并增强：

- `background` 统一消息路由
- `content script` 页面抓取
- `LLM` / `image` 的 provider 抽象
- 本地 storage 作为第一阶段数据层
- 头像资产和头像组件
- 记忆、页面分析、归档、图谱这些能力模块

## 暂时不建议扩大的范围

为了避免发散，当前阶段不建议直接跳到：

- 云端多用户协作
- 远程长期自治运行
- 复杂工作流编排平台
- 多 Agent 协同系统

先把单个插件里的成长闭环做稳，比直接扩大系统更重要。

## 结论

这次新需求不是在原来的“创意插件”上小修小补，而是在把产品重新定义成一个“可投喂、可观察、可调教”的成长型 Agent。

因此工程目标需要从：

> “生成一个好看的结果”

调整为：

> “让 agent 在可控的前提下持续进化，并且每一步都能被看见、被调回、被重用。”
